# 已知遗留与下一轮迭代项

> Stage 7 收尾。这份清单是项目交付时的"诚实面"——所有 MVP 期未做、做了但不完美、或有意延后的事项都在这里。

按优先级 × 影响域分组。

---

## P0 - 阻塞生产发布

### #1 Electron 打包链未跑通
**现状**：dev 模式（`pnpm dev:electron`）可工作，但生产打包未完成自动化。
**影响**：无法分发安装包给真实用户。
**路径**：见 `docs/07-deployment.md` §3，需要：
- 用 esbuild bundle main.ts 为单一 main.cjs
- 配置 electron-builder
- macOS 公证 + Windows 签名

### #2 LLM API Key 明文存储
**现状**：`apiKey` 存在 `.data/db.json`，明文。
**影响**：用户机器被入侵或文件被误分享时密钥泄露。
**路径**：MVP 阶段已通过 `Settings` 模块的接口隔离做好准备：
- 替换 `persistence.putSingleton(settings)` 为 `keytar.setPassword('power-chat', 'llm-api-key', apiKey)`
- 其他 settings 字段仍存 JSON
- 加载时合并 keytar + JSON
**预估工作量**：半天

### #3 Settings UI 未实现
**现状**：只有 IPC/HTTP 接口，没有可视化配置面板。
**影响**：用户首次启动需通过 DevTools 控制台或手动改 `db.json`，体验不可接受。
**路径**：在 prototype 加 `<SettingsDialog>` 组件，挂载到右上角齿轮图标。

## P1 - 影响核心体验

### #4 11 个集成测试 skip 状态（INV-2 / INV-11 协议层守卫）
**现状**：测试覆盖率：33 pass / 11 skip / 0 fail。skip 项主要需要拦截 LLM 客户端的 outbound 请求来验证。
**影响**：INV-2（提炼减熵）和 INV-11（reasoning 隔离）目前只通过代码 review + 类型系统保证，没有运行时断言。
**路径**：把 llm-client 改为依赖注入式（`createLLMClient(config)` 而非全局函数），测试时注入 spy 客户端。

### #5 上下文超长拦截（HTTP 413）未触发
**现状**：模块层已实现（`llm-client.ts::estimateMessagesTokens` + `streamChat` 内的阈值检查），但需要超长输入才触发。
**影响**：超长上下文出错时用户体验不佳（直接报模型错误而非提示提炼）。
**路径**：写一个集成测试构造超长对话验证 413 路径。

### #6 错误恢复 UI 不完整（旅程 2 sad-1, sad-2）
**现状**：模块层支持流式中断保留部分内容（`Message.status='partial'`），但前端没显示错误气泡 + 重试按钮 + "继续生成"选项。
**影响**：网络抖动时用户只看到流式停止，不知如何恢复。
**路径**：在 prototype 的 MessageBubble 组件加错误状态分支。

### #7 Undo / Redo 未实现
**现状**：`docs/02-domain-model.md §1.7` 设计了 ActionLog，但模块层没接入。
**影响**：用户误删节点 / 误移动后只能手动恢复。
**路径**：在 `canvas.ts` 的每个写操作前后捕获 reverse_payload，加一个 `undo-module`，深度 50（Q2-4）。

### #8 节点标题自动生成是 fire-and-forget
**现状**：`conversation.ts::sendMessage` 中 `void updateNodeTitle(...)` 异步调用，不阻塞主流式响应。
**影响**：标题可能更新失败但用户不知道；测试无法确定性验证。
**路径**：改为事件总线模式，UI 订阅 `node:title-updated` 事件实时刷新。

## P2 - 体验优化

### #9 缩放 < 50% 的"超折叠态"色块视图未实现
**现状**：当前 < 50% 缩放节点仍渲染完整内容（性能可能下降）。
**影响**：30+ 节点时俯瞰视角卡顿。
**路径**：在 Node.tsx 检测 `zoom < 0.5` 时切换为简化 div。

### #10 100 节点软上限的 toast 提示未实现
**现状**：模块层不限制创建数量，前端没监测达到 100 节点时弹提示（Q5-1 选 B）。
**影响**：用户撞到性能边界时无引导。

### #11 节点超折叠态卡片的 spinner 指示器
**现状**：折叠态卡片仅显示 ●，不是清晰的旋转 spinner（A1 决策）。
**影响**：流式期间折叠节点的状态指示视觉强度不够。

### #12 Tab 键切换最近节点 + 平滑滚动（A5、Q7-4）
**现状**：未实现键盘导航。
**影响**：键盘可达性不达标。

### #13 多选只支持 Shift+点击，不支持框选（B1）
**现状**：MVP 决策不做框选。
**影响**：选中超过 5 个节点时操作繁琐。

### #14 Token 估算不区分模型（C3 决策）
**现状**：单一保守系数（中文 ×0.7 / 英文 ×0.4）。
**影响**：不同模型实际 token 比例不同，估算可能偏差 20-30%。
**路径**：二阶段集成 `tiktoken` 或各模型官方 tokenizer。

## 关于 MVP 期"成功"的承认

PRD §10 提出三个待验证假设：
1. 用户会主动选择分支
2. 提炼出的纲要对后续思考有用
3. 对话→提炼→更高层对话的循环成立

**当前架构没有自带数据证明这三件事**——它只把这些动作做出来了。验证需要 Stage 7 之后的"种子用户观察"环节，这是产品工作不是工程工作。

工程上 MVP 已经达到的事：
- ✅ 单一可运行的桌面应用（Electron dev 模式）
- ✅ AI 模块解耦（PRD §1.5 原则 A、B 在工程上完全实现）
- ✅ 4 个核心动作贯通（种节点 / 分支 / 提炼 / 继续）
- ✅ 12 条不变量在代码里有明确锚点
- ✅ 端到端契约一致（OpenAPI → mock-server → IPC → renderer 全链路验证过）
- ✅ 33 个集成测试守住关键行为

工程上未达到的事：
- ❌ 生产可分发（打包链未完成，#1）
- ❌ 安全合规（API Key 明文，#2）
- ❌ 完整 UI（设置面板、错误恢复、Undo，#3 #6 #7）
- ❌ 性能验证（100 节点未实测，#9）

这些是 MVP-to-Alpha 的工作，不是 MVP-internal 的回头事项。
