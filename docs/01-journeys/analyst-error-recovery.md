# 旅程：分析师 - AI 调用与配置错误恢复

## 元信息
- 角色：独立行业分析师 林然（同 journey-1）
- 意图：在研究过程中遭遇 LLM 调用相关的各种异常（API Key 失效、网络问题、上下文超长、provider 切换），能不丢失工作进度地恢复
- 优先级：P0（决定产品能否在不稳定网络/多 LLM 环境下被信任）
- 来源：Q3-2、Q3-3、Q3-5、Q7-2 + PRD §9（LLM 接入）

## 前置状态
- 应用已配置 baseURL = `https://api.deepseek.com/v1`，模型 `deepseek-reasoner`
- 画布上已有 8 个节点，林然正在做"新茶饮出海"研究的第二天工作
- 当前活跃节点是一个对话节点 N5（已有 6 条消息）

## Sad Path 1：API Key 失效（用户在公司临时换了账号）

1. 林然在节点 N5 输入新问题：
   ```
   假设我们要做一个 12 个月的供应链建设路径图，按月拆解关键里程碑。
   ```
   按 Enter 提交

2. 应用调起 LLM 流式请求，但 1.2 秒后服务端返回 401 Unauthorized
   - 流式响应未开始任何 token 输出
   - 节点内出现红色错误气泡：
     > 调用失败：API Key 无效或已过期 (HTTP 401)
     >
     > [打开设置]  [重试]  [关闭]
   - 用户的输入消息**保留在节点中**（已 commit），AI 那条空回复显示为错误状态（不算一条消息，不进入历史）

3. 林然点击 "打开设置"
   - 应用打开设置面板（modal 或独立页面），定位到 "LLM 配置" 区
   - 当前 baseURL / 模型 / Key 都可见，Key 字段显示 `sk-•••••••••3a9f`
   - 林然修改 Key 为新值，点击 "保存并测试"
   - 应用对当前 baseURL 发一次轻量 ping 调用（如 `models` 列表），返回 200 → 显示绿色 "✓ 配置可用"
   - 关闭设置

4. 林然回到画布，节点 N5 的红色错误气泡依然在
   - 林然点击气泡上的 "重试"
   - 应用用新 Key 重新发送相同请求
   - AI 流式回复正常返回，错误气泡消失
   - 整个失败-恢复过程**没有丢失任何对话内容**，节点上下文连续

## Sad Path 2：流式输出中途网络抖动

5. 林然换了一个节点 N6 提问，AI 开始流式回复
6. AI 输出到约 230 字时（占预期内容 60%），WiFi 切换导致连接中断
   - 应用 SSE 连接断开（在 12 秒读取超时内未收到新事件）
   - 已收到的 230 字**保留**在节点中，不擦除
   - 节点内出现黄色警告气泡（不是红色错误）：
     > 网络中断，已收到部分内容已保留。
     >
     > [继续生成]  [重新发送]  [保留并关闭]
   - 三选项含义：
     - **继续生成**：用 "请基于刚才被截断的回复继续往下写，从 '...XX' 这里接着" 这个补充指令重新调起 LLM（产品层封装，AI 不知道）
     - **重新发送**：丢弃 230 字，重新发起完整请求
     - **保留并关闭**：把 230 字当成完整回复入库，不再调用 AI

7. 林然选 "继续生成"
   - 应用拼接补全请求发送
   - AI 从断点附近继续输出剩余内容，与原 230 字在视觉上无缝拼接
   - 节点最终保存一条完整 assistant 消息（230 字 + 续接内容），但标记位 `was_resumed = true`（用于后续观察是否影响标题生成质量）

## Sad Path 3：上下文超长，前端拦截

8. 林然在另一个深度分支节点 N8 上已经讨论了 22 轮，加上父链一共积累 ~48K token
9. 他再输入一条长问题（约 800 字的复杂场景假设），按 Enter 提交
10. 应用在发送前做 token 估算：
    - 计算方式：当前节点完整对话 + 父链所有对话 + 当前输入 + 预留的 reasoning + 预留的 max_tokens
    - 估算结果 56K，超过 deepseek-reasoner 上限 64K 的 80% 阈值（51K）
11. 前端**不发送**，节点内出现黄色提示气泡：
    > 当前节点的对话上下文已接近模型容量上限（估算 56K / 64K）。
    > 继续发送可能被截断或失败。
    >
    > 建议：先选中本节点（和相关节点）做一次"提炼"，
    > 在提炼节点上继续讨论可显著降低上下文。
    >
    > [仍然发送]  [我去提炼]  [关闭]
12. 林然点 "我去提炼"
    - 提示气泡关闭
    - 应用自动选中当前节点 N8 + 它的父节点（基于父链关系），多选状态可视化
    - 顶部工具栏 "提炼 (2)" 自动高亮（脉动一次吸引注意）
    - 林然点击"提炼"，输入意图问题，AI 产出约 380 字纲要，提炼节点 N9 生成
    - 林然点击 N9 继续深入，N9 的上下文 = 纲要内容（~600 token）+ 新对话
    - 之前那条 800 字输入暂存为草稿放在 N8 输入框（用户可手动复制粘贴到 N9 重新提交，MVP 不自动转移）

## Sad Path 4：用户切换 LLM provider（中途换模型）

13. 林然觉得 deepseek-reasoner 速度慢，想切到本地 Ollama 上的 qwen3
14. 打开设置：
    - 修改 baseURL = `http://localhost:11434/v1`
    - 修改模型 = `qwen3:14b`
    - Key 字段：填一个占位符（Ollama 不需要 Key，但 OpenAI 兼容客户端要求字段非空）
    - 思考模式开关：保持开启
    - 点击 "保存并测试" → 测试通过
15. 林然回到任意节点继续对话
    - 这次的请求发到 localhost
    - 但 qwen3:14b 不返回 reasoning 字段（不支持思考模式）
    - **静默降级**：节点上方不显示思考区域，仅显示普通 assistant 回复
    - 不报错，不弹提示（避免噪音；用户主动切换前已知风险）
16. 林然回到设置切回 deepseek-reasoner
    - 后续节点的对话又开始有思考区域

## Sad Path 5：首次启动未配置 LLM

17. 假设是另一台机器/全新安装：林然双击应用图标
18. 应用首次启动流程：
    - 弹一次性隐私提示（Q7-2）：
      > 你的对话内容会被发送到你配置的 LLM 端点。
      > 应用本身不收集任何数据，所有画布数据保存在本地。
      > [我已知悉，进入应用]
    - 用户勾选并进入主界面
    - 主界面是空白画布 + 一个**居中提示卡片**：
      > 还没有配置 LLM 端点。
      > 点击右上角设置图标，填入 baseURL、模型、API Key 后即可开始。
      >
      > [打开设置]
19. 林然点击 "打开设置"，配置流程同步骤 14
20. 配置完成回到画布，提示卡片消失，可正常双击空白处开始
21. 在配置之前，所有"双击空白创建节点"的动作仍可执行，但节点输入提交时会拦截：
    > 请先完成 LLM 配置 → [打开设置]
    - 这避免用户误以为应用坏了

## 边界
- 错误气泡里的"重试"，最大自动重试次数 0（不自动重试，避免无意义反复消耗）
- 任何 LLM 错误**不会**回滚用户已输入的消息——用户的话总是保留
- 如果在思考模式流式中途出错，已收到的 reasoning 内容也保留（与正文同样保留策略）
